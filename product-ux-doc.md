# OtterQuant / Quandora 产品介绍与 UX 文档

版本：v1.0  
适用范围：当前本地预览版本 `codex/gamification2.0`  
文档目的：统一产品叙事、核心用户体验、Agent 连接流程、界面信息架构与后续验收标准。

## 1. 产品概述

OtterQuant / Quandora 是一个面向量化研究者、策略开发者与 AI Agent 用户的 AI 因子挖掘平台。产品以轻量游戏化的“钓鱼”体验作为入口，将复杂的因子探索、资产积累、策略发现、Agent 连接和个人账户管理包装成更易理解、可持续使用的互动系统。

产品的核心目标不是把专业量化流程娱乐化到失真，而是用游戏化降低进入门槛，用 AI Agent 提升研究效率，用清晰的账户与连接管理保证高级用户可以稳定地把平台接入自己的工作流。

### 1.1 一句话介绍

OtterQuant / Quandora 是一个通过 AI Agent 和游戏化交互帮助用户发现、管理与使用量化因子的研究平台。

### 1.2 核心价值

- 降低因子研究的启动成本：用户不需要一开始面对复杂参数表，而是从可理解的“抛竿、收获、图鉴、市场”开始。
- 提升持续使用动力：通过鱼额、鱼篓、图鉴、排行榜等反馈机制，让探索行为有即时回报。
- 支持多种 AI 接入方式：用户既可以在网页上快速授权使用，也可以在自己的 Agent 客户端中安装插件。
- 兼顾新手与专业用户：新手可以在网页中完成轻量连接，开发者可以通过 API Key 或本地 Agent 插件接入更完整的工作流。

## 2. 目标用户

### 2.1 量化研究者

这类用户关注因子质量、策略验证效率、历史数据表现与可复用研究成果。他们希望平台能帮助自己更快筛选有价值的研究方向，并减少重复整理工作。

核心需求：

- 快速发现潜在因子。
- 管理已有因子与策略线索。
- 查看历史成果与表现反馈。
- 将 AI 辅助能力接入日常研究流程。

### 2.2 策略开发者

这类用户偏工程化，关注 API Key、本地 Agent、插件安装、配置路径和连接稳定性。他们需要平台提供明确的接入方式和可控的授权状态。

核心需求：

- 明确知道当前连接方式。
- 能快速获取 API Key 或授权码。
- 能复制官方安装命令或查看手动安装路径。
- 能断开、切换或恢复连接。

### 2.3 AI Agent 用户

这类用户可能已经在使用 Codex、Claude Code、OpenClaw 等 Agent，希望把 OtterQuant 能力接入自己的 Agent 环境。

核心需求：

- 在多个 Agent 中同时安装插件。
- 知道每个 Agent 的连接状态。
- 一键断开全部 Agent 连接。
- 清楚区分“网页上用”和“自己的 Agent 上用”两类模式。

### 2.4 轻量探索用户

这类用户可能还不了解量化研究，只是被游戏化体验吸引。他们需要低压力地体验产品价值。

核心需求：

- 第一屏能立即理解可以做什么。
- 交互结果清晰、有反馈。
- 不被专业术语劝退。
- 在需要时自然进入设置、钱包、市场或图鉴。

## 3. 产品定位

OtterQuant / Quandora 处在“AI 研究工具”和“轻量游戏化产品”的交叉位置。

它不是纯工具后台，也不是纯游戏。它的设计重点是让复杂工作流具有可见进度、可收集成果和可持续探索的节奏。

### 3.1 产品关键词

- AI 因子挖掘
- 量化研究
- Agent 插件
- 授权连接
- 游戏化探索
- 因子资产
- 图鉴收集
- 市场流通
- 排行榜激励

### 3.2 不做什么

- 不把所有专业能力塞进第一屏。
- 不用营销页替代真实功能体验。
- 不让用户在未理解连接方式前进入复杂配置。
- 不允许网页模式下同时连接多个 Agent。
- 不把本地 Agent 插件安装流程隐藏在模糊说明里。

## 4. 产品信息架构

当前体验以游戏主界面为中心，所有主要功能通过顶部/侧边功能入口进入。

### 4.1 一级入口

- 鱼塘：核心探索/抛竿场景，承载主要游戏化操作。
- 鱼市场：因子或资源的流通、兑换、购买与出售入口。
- 图鉴：用户已发现内容的收藏、回顾与完成度展示。
- 钱包：资产、鱼额、账户余额与可能的交易记录入口。
- 排行榜：展示用户竞争关系、探索成就与长期激励。
- 设置：语言、个人资料、Agent 连接、桌面伴侣下载等配置入口。

### 4.2 设置模块

设置模块承担用户身份、语言偏好与 Agent 接入管理。

主要分区：

- 通用设置：语言、个人资料、邮箱、密码。
- Agent 设置：连接方式选择、连接服务管理、授权/安装流程、连接状态反馈。
- Buddy 下载：桌面伴侣下载入口，用于同步主界面钓鱼状态。

## 5. 核心体验原则

### 5.1 先体验，再解释

首页优先呈现可操作的钓鱼体验，而不是长篇产品介绍。用户进入后应能直接看到资产、入口和主操作。

### 5.2 复杂配置分步骤暴露

Agent 连接流程必须分层呈现：

1. 先选择使用环境。
2. 再选择授权方式或 Agent 类型。
3. 最后进入具体配置、反馈和管理状态。

### 5.3 状态必须显性

连接状态只使用清晰的“未连接”和“已连接”，避免“尚未授权”“等待连接”等多重状态混淆。必要时通过提示条解释为什么某个操作不可用。

### 5.4 禁用状态也要有解释

网页模式下存在已连接 Agent 时，其余 Agent 进入禁用状态。点击禁用项时，不应沉默失败，而应提示用户先断开已连接 Agent。

### 5.5 管理动作靠近问题出现处

当用户因为已有连接而无法切换方式或连接新服务时，“一键断开全部”应出现在提示条旁边，跟随提示显示/隐藏，减少寻找成本。

## 6. 核心用户流程

## 6.1 首次进入主界面

目标：让用户在 5 秒内理解这是一个可操作的量化探索空间。

用户看到：

- 当前资产：例如鱼额、钱包数值。
- 功能入口：鱼塘、鱼市场、图鉴、钱包、排行榜、设置。
- 主操作：抛竿、自动抛竿、鱼篓。

关键 UX 要求：

- 主操作按钮必须比次级入口更突出。
- 资产信息应稳定占位，不因数字变化造成明显布局跳动。
- 图标和标签配对展示，降低认知成本。
- 不在第一屏放置大段说明性文案。

## 6.2 抛竿与收获流程

目标：让用户通过一次操作获得即时反馈，并形成继续探索的动机。

建议流程：

1. 用户点击“抛竿”。
2. 系统展示进行中状态，例如按钮反馈、动效或等待态。
3. 系统返回收获结果。
4. 结果进入鱼篓或图鉴。
5. 用户可继续抛竿、查看鱼篓、进入图鉴或市场。

关键状态：

- 默认：抛竿按钮可用。
- 进行中：按钮不可重复触发，显示明确等待反馈。
- 成功：展示收获内容、稀有度或价值。
- 失败/空竿：给出轻量反馈，不打断主流程。

## 6.3 自动抛竿流程

目标：满足长期探索和挂机式体验。

建议流程：

1. 用户点击“开启自动抛竿”。
2. 系统切换为自动状态。
3. 用户可随时关闭。
4. 自动过程中的收获进入鱼篓或图鉴。

关键 UX 要求：

- 自动状态必须显性，避免用户误以为系统无响应。
- 关闭入口必须与开启入口位置一致。
- 资产变化和收获反馈应节制，不频繁弹窗干扰。

## 6.4 钱包流程

目标：让用户理解资产来源、余额变化和可用资金。

建议内容：

- 当前余额。
- 鱼额或平台内积分。
- 最近交易/变化记录。
- 充值、提现或兑换入口，如果当前版本支持。

关键 UX 要求：

- 金额展示要对齐、易扫读。
- 重要资产变动需要提供来源说明。
- 危险操作需要确认。

## 6.5 鱼市场流程

目标：让用户处理收获物、资源或因子资产。

建议内容：

- 可出售/可购买项目。
- 稀有度、价格、收益或用途。
- 筛选与排序。
- 确认交易。

关键 UX 要求：

- 市场信息密度可以高于首页，但要保持清晰层级。
- 价格、数量、操作按钮不能混在一起。
- 交易前后应有明确反馈。

## 6.6 图鉴流程

目标：让用户看到探索进度和长期目标。

建议内容：

- 已解锁项目。
- 未解锁占位。
- 完成度。
- 分类、稀有度、来源说明。

关键 UX 要求：

- 未解锁内容可以保留悬念，但不要影响已解锁内容阅读。
- 已收集项目应有明确状态。
- 图鉴是长期留存模块，应强调进度感。

## 6.7 排行榜流程

目标：提供社交比较和长期激励。

建议内容：

- 用户排名。
- 关键指标，例如总收获、稀有收获、策略价值或活跃度。
- 周榜、总榜或好友榜。

关键 UX 要求：

- 当前用户位置应易找到。
- 榜单指标需要解释清楚，避免用户不知道如何提升排名。
- 避免排行榜对新用户造成过强挫败感，可提供分层榜单。

## 7. Agent 连接 UX

Agent 设置是当前版本最重要的复杂流程，需要保证规则清晰、状态稳定、可恢复。

## 7.1 Agent 连接总规则

第一步：选择连接方式。

可选方式：

- 我要在网页上用。
- 我要在自己的 Agent 上用。

规则：

- 连接方式只允许二选一。
- 如果需要修改方式，必须先断开当前连接。
- 当已有连接阻止切换时，展示提示条。
- 提示条旁展示“一键断开全部”，并跟随提示显示/隐藏。

## 7.2 网页上用

适用场景：用户希望在网页内快速使用 AI 能力，不想安装本地插件。

可连接服务：

- Codex。
- OpenRouter。

连接限制：

- 仅允许连接 1 个 Agent。
- 当存在已连接 Agent 时，其余 Agent 进入禁用状态。
- 点击禁用 Agent 时，提示：“请先断开已连接的 agent，再进行新的连接。”

状态类型：

- 未连接。
- 已连接。

流程：

1. 用户选择“我要在网页上用”。
2. 系统进入可连接服务列表。
3. 用户选择 Codex 或 OpenRouter。
4. 系统展示对应授权方式。
5. 用户完成授权或 API Key 配置。
6. 连接成功后回到弹窗，显示当前连接状态。
7. 用户可检查状态、断开连接或切换连接方式。

### 7.2.1 Codex 网页连接

建议授权方式：

- 授权码。
- API Key。

API Key 辅助链接：

- Get API Key from OpenAI。

关键 UX 要求：

- 授权码与 API Key 是选择关系，不应同时展开造成干扰。
- API Key 输入区域高度应 hug 内容，不要出现过高空白。
- 成功后回到管理视图，展示 Codex 已连接。

### 7.2.2 OpenRouter 网页连接

OpenRouter 只保留 API Key 配置。

弹窗中应删除：

- “API Key”选项标题。
- “使用 API Key 直接调用，适合开发者。”选项描述。

保留：

- API Key 输入。
- Get API Key from OpenRouter 链接。
- 验证并连接按钮。

关键 UX 要求：

- 因 OpenRouter 没有多授权方式选择，界面不应再呈现“选项卡/单选项”的视觉暗示。
- 表单应直接出现在内容区，减少框套框。
- 容器高度跟随内容，不额外撑高。

## 7.3 在自己的 Agent 上用

适用场景：用户希望在本地 Agent 客户端中安装插件，接入 OtterQuant 能力。

可连接 Agent：

- Codex。
- Claude Code。
- OpenClaw。

连接限制：

- 允许同时连接多个 Agent。
- 用户可以分别查看每个 Agent 的安装指引与连接状态。
- 用户可一键断开全部。

流程：

1. 用户选择“我要在自己的 Agent 上用”。
2. 系统展示可连接 Agent 列表。
3. 用户选择 Codex、Claude Code 或 OpenClaw。
4. 系统提供插件安装指引。
5. 用户选择自动安装或手动安装。
6. 自动安装提供官方命令行，支持复制。
7. 手动安装提供官方路径指引。
8. 连接成功后展示已连接状态。
9. 用户可断开单个 Agent 或一键断开全部。

### 7.3.1 自动安装

信息结构：

- Agent 名称。
- 官方安装命令。
- 复制按钮。
- 安装完成后的检查按钮。

关键 UX 要求：

- 命令区域使用等宽字体。
- 复制成功需要即时反馈。
- 不把长命令挤进普通段落。

### 7.3.2 手动安装

信息结构：

- 插件路径。
- 配置文件位置。
- 简短步骤。
- 检查连接状态按钮。

关键 UX 要求：

- 每一步只表达一个动作。
- 路径和命令需要可复制。
- 错误状态需要告诉用户下一步做什么。

## 8. Agent 设置界面结构

Agent 设置建议由以下区域组成：

1. 顶部连接方式状态栏。
2. 阻断提示条。
3. 可连接服务列表。
4. 连接/授权弹窗。
5. 授权弹窗示意。
6. Buddy 下载入口。

### 8.1 顶部连接方式状态栏

展示当前模式：

- 在网页上用。
- 在自己的 Agent 上用。
- 未选择连接方式。

提供操作：

- 切换连接方式。

当已有连接时，点击切换不直接进入新模式，而是展示提示条。

### 8.2 阻断提示条

提示类型：

- 切换方式阻断：请先断开当前连接，再切换连接方式。
- 连接新服务阻断：请先断开已连接的 agent，再进行新的连接。

提示条内操作：

- 一键断开全部。

显示规则：

- 仅在存在阻断时显示。
- 阻断解除后隐藏。
- 不应在没有阻断时常驻。

### 8.3 可连接服务列表

每个服务行包含：

- Agent 标识。
- Agent 名称。
- 状态：未连接 / 已连接。
- 操作按钮：连接、检查、断开。

网页模式：

- Codex。
- OpenRouter。
- 已连接一个服务后，其他服务禁用。

自己的 Agent 模式：

- Codex。
- Claude Code。
- OpenClaw。
- 可同时连接多个。

### 8.4 连接弹窗

弹窗目标：聚焦完成单个连接任务。

弹窗结构：

- 标题：连接 Agent 或连接具体服务名。
- 内容：授权方式、API Key 表单或插件安装指引。
- 操作：验证并连接、复制命令、检查连接、取消或关闭。

视觉要求：

- 减少框套框。
- 如果只有一种连接方式，不展示选项头和说明。
- 表单区域高度跟随内容。
- 主要操作固定在内容结束处，避免用户找不到提交按钮。

## 9. 状态与反馈规范

### 9.1 连接状态

只使用：

- 未连接。
- 已连接。

不建议混用：

- 尚未授权。
- 等待连接。
- 已授权但未连接。

如果后续确实需要更多状态，应扩展为明确状态机，而不是在同一位置混合文案。

### 9.2 按钮状态

按钮应覆盖：

- 默认。
- Hover。
- Pressed。
- Disabled。
- Loading。
- Success。
- Danger。

禁用按钮不应只依赖颜色变化，还需要：

- 禁用光标。
- 降低透明度或对比度。
- 点击时给出解释，如果业务需要用户理解原因。

### 9.3 成功反馈

成功连接后：

- 回到管理视图。
- 对应服务显示“已连接”。
- 可以进行“检查”和“断开”。
- 如果是网页模式，其他服务进入禁用。

### 9.4 错误反馈

错误反馈需要包含：

- 发生了什么。
- 为什么可能发生。
- 用户下一步可以做什么。

示例：

- API Key 无效：请检查 API Key 是否完整，或重新从官方页面生成。
- 插件未检测到：请确认安装命令已执行，并重启对应 Agent 客户端。
- 无法切换方式：请先断开当前连接，再切换连接方式。

## 10. 视觉与布局规范

### 10.1 整体风格

当前产品使用像素湖边游戏场景作为主体验，设置弹窗与功能面板应保持轻量、柔和、可扫读。

关键词：

- 清爽。
- 轻游戏化。
- 明确状态。
- 低压探索。
- 专业能力隐藏在清晰流程之后。

### 10.2 布局原则

- 主界面优先保证操作可见。
- 设置弹窗内部避免多层卡片嵌套。
- 重复项可以使用卡片或行布局。
- 表单不应被包在过多视觉容器中。
- 警告和管理动作应在同一视线范围内。

### 10.3 文案层级

建议层级：

- 页面标题：说明当前模块。
- 区域标题：说明当前任务。
- 状态文案：短、明确。
- 辅助说明：只在必要时出现。
- 操作按钮：使用动词。

### 10.4 图标使用

- 功能入口使用图标+文字。
- Agent 标识使用各自头像或缩写。
- 外链使用外跳图标。
- 复制、检查、断开等工具动作优先使用熟悉图标辅助。

## 11. 文案规范

### 11.1 语气

文案应清楚、直接、轻量，不使用过度营销语。

推荐：

- 选择连接方式。
- 请先断开当前连接，再切换连接方式。
- 请先断开已连接的 agent，再进行新的连接。
- 验证并连接。
- 一键断开全部。

避免：

- 您的伟大 AI 旅程即将开始。
- 授权失败，请重试。
- 尚未授权 · 等待连接。

### 11.2 中英文混排

保留产品与服务名：

- Codex。
- OpenRouter。
- Claude Code。
- OpenClaw。
- API Key。
- Agent。

中文说明中不强行翻译这些专有名词。

### 11.3 操作文案

按钮文案应短：

- 连接。
- 断开。
- 检查。
- 复制。
- 下载。
- 下一步。
- 验证并连接。

危险操作使用明确结果：

- 确认断连。
- 一键断开全部。

## 12. 可访问性要求

### 12.1 键盘可用

必须支持：

- Tab 切换主要控件。
- Enter / Space 激活按钮和可选卡片。
- Esc 关闭弹窗。
- 焦点不应逃出当前弹窗。

### 12.2 语义

- 设置弹窗使用 dialog。
- 设置分类使用 tab / tablist / tabpanel。
- 可点击禁用行如果需要解释，应具备 `aria-disabled`。
- 关闭按钮需要明确 aria-label。

### 12.3 颜色对比

- 状态文字与背景需要保持可读。
- 警告提示不能只靠颜色表达。
- 禁用状态需要配合透明度、光标或文字提示。

## 13. 响应式要求

### 13.1 桌面端

重点：

- 保持主操作区域稳定。
- 设置弹窗可容纳 Agent 列表和连接表单。
- 长命令、API Key 输入和外链不溢出。

### 13.2 移动端

重点：

- 功能入口可分行展示。
- 设置弹窗内容应纵向流动。
- Agent 服务行可以堆叠。
- 按钮保持足够点击面积。
- 警告条内“一键断开全部”允许换行，但不能遮挡正文。

## 15. 验收清单

### 15.1 产品主界面

- 首页不展示营销落地页，而是直接展示可操作体验。
- 用户能看到资产、入口和主操作。
- 抛竿按钮状态清晰。
- 鱼篓数量变化不会造成布局跳动。

### 15.2 Agent 连接方式

- 第一步先选择连接方式。
- 连接方式只允许二选一。
- 已有连接时不能直接切换方式。
- 阻断提示条出现时，同步显示“一键断开全部”。
- 阻断解除后提示条隐藏。

### 15.3 网页模式

- 只展示 Codex、OpenRouter。
- 只允许连接一个 Agent。
- 已连接一个 Agent 后，其余 Agent 禁用。
- 点击禁用 Agent 显示提示。
- 状态只显示“未连接”或“已连接”。

### 15.4 OpenRouter

- 只展示 API Key 表单。
- 不展示 API Key 选项标题。
- 不展示“使用 API Key 直接调用，适合开发者。”描述。
- 链接文案为 Get API Key from OpenRouter。
- 表单容器高度跟随内容。

### 15.5 Codex 网页连接

- 支持授权码/API Key。
- API Key 链接文案为 Get API Key from OpenAI。
- 成功后回到连接管理状态。

### 15.6 自己的 Agent 模式

- 只展示 Codex、Claude Code、OpenClaw。
- 允许同时连接多个 Agent。
- 提供自动安装命令。
- 提供手动安装路径指引。
- 支持一键断开全部。

## 16. 后续迭代建议

### 16.1 短期

## 17. 术语表

- Agent：可执行用户任务的 AI 客户端或智能体，例如 Codex、Claude Code、OpenClaw。
- 网页上用：在 OtterQuant / Quandora 网页内直接完成授权并使用 AI 能力。
- 自己的 Agent 上用：在用户本地 Agent 客户端安装插件后使用平台能力。
- API Key：用于授权访问第三方服务的密钥。
- 授权码：用于网页授权流程的一次性或短期凭证。
- 鱼额：平台内资产或积分的游戏化表达。
- 鱼篓：当前收获的临时容器。
- 图鉴：已发现内容和进度的集合。
- 鱼市场：资源交易或流通场景。
